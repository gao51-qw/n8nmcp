# Dedicated Mail Domain Deployment Design

## Goal

Configure the existing production mail stack and the root Next.js application
to use all of the following consistently:

- Sender mailbox: `server@n8nworkflow.com`
- Mail domain: `n8nworkflow.com`
- SMTP submission hostname: `server.n8nworkflow.com`

The change must preserve the existing BillionMail installation, keep unrelated
projects and mailboxes untouched, and finish with evidence of real external
delivery rather than SMTP queue acceptance alone.

## Current State and Constraints

- The VPS already hosts BillionMail. A previous production exercise verified
  SMTP AUTH, ports 465 and 587, and a trusted TLS certificate after certificate
  synchronization was configured.
- The VPS provider blocks direct outbound SMTP delivery on port 25. Messages to
  Gmail and Outlook remained deferred even though local authentication, TLS,
  DKIM processing, and queueing succeeded.
- External delivery therefore requires an authenticated SMTP relay. The prior
  exercise verified that Resend alternate ports 2465 and 2587 were reachable.
- At design time, `server.n8nworkflow.com` has no public DNS record and
  `n8nworkflow.com` has no MX record. DNS and relay-domain verification are hard
  deployment gates.
- The repository has a large dirty working tree. Implementation must preserve
  every unrelated staged, modified, deleted, and untracked path.
- Secrets must remain in root-readable production environment files or the
  mail-system secret store. They must never be committed or printed in logs.

## Chosen Architecture

Reuse the existing BillionMail runtime and add an isolated mail domain and
mailbox for this project. Do not deploy a second mail stack and do not retain
the old `adsmcpserver.com` mailbox as the public identity.

The resulting flow is:

1. The application and Supabase Auth authenticate to
   `server.n8nworkflow.com:465` with implicit TLS. Port 587 with STARTTLS is the
   documented fallback.
2. BillionMail accepts authenticated submissions for
   `server@n8nworkflow.com`.
3. BillionMail routes outbound messages through the configured external relay,
   because direct port-25 delivery is unavailable.
4. Public recipients validate SPF, DKIM, and DMARC for `n8nworkflow.com`.

## DNS and TLS

Production DNS must contain:

- An unproxied A record for `server.n8nworkflow.com` pointing to the current
  mail VPS.
- An MX record for `n8nworkflow.com` targeting `server.n8nworkflow.com`.
- The SPF record required by the selected relay and local mail architecture.
- The DKIM selector and public key generated by BillionMail or required by the
  relay configuration.
- A DMARC record beginning in monitoring mode (`p=none`) with aggregate reports
  directed only to an explicitly provisioned mailbox. Policy tightening is out
  of scope for this deployment.

Cloudflare proxying must remain disabled for SMTP and MX hostnames. The
BillionMail SMTP services must present a publicly trusted certificate whose SAN
includes `server.n8nworkflow.com`. Certificate renewal synchronization must be
extended to the new hostname before production SMTP configuration changes.

## Mail-System Changes

- Add `n8nworkflow.com` as a BillionMail mail domain.
- Create the dedicated mailbox `server@n8nworkflow.com` with a generated,
  high-entropy password and no administrative privileges.
- Preserve all pre-existing domains, mailboxes, queues, and relay settings.
- Bind the trusted certificate for `server.n8nworkflow.com` to Postfix/Dovecot
  using the existing certificate synchronization mechanism.
- Configure or verify the authenticated external relay on a provider-supported
  alternate port. Relay credentials must not appear in the repository.
- Refuse activation if SMTP AUTH, hostname verification, or relay authentication
  fails.

## Application and Supabase Changes

The root Next.js application remains the only application deployment target.
Implementation will:

- Replace public contact and support mail links with
  `server@n8nworkflow.com`.
- Set the production support-notification sender to
  `server@n8nworkflow.com` through runtime configuration.
- Set the public security contact to `server@n8nworkflow.com` where the existing
  `NEXT_PUBLIC_SECURITY_EMAIL` contract is used.
- Update deployment templates and documentation with non-secret SMTP host,
  port, TLS, and sender values.
- Configure production Supabase Auth to use
  `server.n8nworkflow.com:465`, the dedicated mailbox username, implicit TLS,
  and a securely stored password.

The application currently sends support notifications through the Resend API.
That transport may remain in place as long as its verified sender is
`server@n8nworkflow.com`. SMTP hostname changes apply to Supabase Auth and any
SMTP clients; no redundant application SMTP transport will be introduced.

## Deployment Sequence and Rollback

1. Capture a redacted inventory of current DNS, BillionMail domains, mailboxes,
   certificates, relay status, Supabase Auth mail variables, and application
   image/version.
2. Add and verify DNS records, relay-domain ownership, and the trusted TLS
   certificate.
3. Back up BillionMail, Supabase Auth, and application environment
   configuration before mutation.
4. Add the mail domain and mailbox, then verify TLS and SMTP AUTH without
   sending externally.
5. Update Supabase Auth and application runtime configuration. Validate Compose
   before recreating only the affected services.
6. Deploy the verified root Next.js image through the repository's VPS
   Docker/aaPanel path. Do not introduce Vercel, Cloudflare Worker, or legacy
   Express deployment targets.
7. Run external delivery and authentication acceptance tests.

If any gate fails, stop progression and restore the most recent configuration
backup for the affected service. DNS additions may remain only when they do not
redirect production traffic to an unverified endpoint. Existing mail domains
and unrelated services are never rollback targets.

## Validation and Acceptance Criteria

Repository verification must include focused tests for email constants and
deployment contracts, followed by lint, type checking, the relevant test suite,
and a production build.

Production acceptance requires fresh evidence for every item below:

- `server.n8nworkflow.com` resolves to the intended VPS and is not proxied.
- SMTP on port 465 presents a trusted certificate valid for the hostname.
- `server@n8nworkflow.com` authenticates successfully without exposing its
  password.
- SPF, DKIM, and DMARC records resolve publicly.
- A real registration email sent through Supabase Auth reaches an external
  Gmail inbox, rather than merely entering the Postfix queue.
- The confirmation link succeeds and the confirmed user can sign in.
- A support notification reaches an external inbox with From set to
  `server@n8nworkflow.com` and passes SPF/DKIM/DMARC evaluation.
- The root Next.js application and affected Supabase services are healthy after
  deployment.
- No unrelated working-tree path, VPS project, mailbox, or container changed.

If relay credentials or DNS control are unavailable, repository changes may be
completed and locally verified, but production deployment must be reported as
blocked rather than complete.

## Out of Scope

- Deploying a second BillionMail instance.
- Migrating or deleting existing `adsmcpserver.com` mailboxes.
- Opening direct outbound port 25 or weakening TLS verification.
- Adding a new application SMTP library when the existing Resend API transport
  satisfies support notifications.
- Tightening DMARC beyond monitoring mode during the initial migration.
